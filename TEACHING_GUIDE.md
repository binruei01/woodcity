# 探索嘉義木都文化與在地導覽 AI 查詢平台：專案實作教學指南

這份指南旨在幫助老師帶領學生理解並實作「探索嘉義木都文化與在地導覽 AI 查詢平台」網站。本專案結合了現代網頁開發技術與生成式 AI (Gemini)，是一個標準的 PBL (專題式學習) 實作案例。

---

## 一、 技術架構 (Technology Stack)

本網站採用業界標準的現代化工具開發：

1.  **React 18**: 全球最流行的前端框架，使用「組件化」思維開發網頁。
2.  **TypeScript**: 為 JavaScript 加入型別定義，減少程式錯誤，適合初學者建立良好編碼規範。
3.  **Vite**: 極速的前端開發補給工具，負責處理程式碼編譯與預覽。
4.  **Tailwind CSS**: 實踐式的 CSS 框架，透過「類名」直接在 HTML 中進行設計，大幅縮減樣式編寫時間。
5.  **Framer Motion**: 強大的動畫庫，用於處理卡片進場、問答視窗切換等流暢動畫。
6.  **Lucide React**: 輕量且標誌清晰的向量圖示庫，提供直觀的視覺引導。

## 二、 功能模組 (Functional Modules)

網站由以下六大組件構成，教學時可引導學生逐一完成：

1.  **導覽列 (Navigation)**：
    *   固定於頂部，顯示專案名稱。
    *   內建「系統診斷鈕」，幫助開發者即時確認 API 連線狀態。
2.  **核心英雄區 (Hero Section)**：
    *   大標題與副標題，使用視覺強烈的配色營造第一印象。
    *   強調 PBL 教學計畫團隊資訊。
3.  **知識圖卡組 (Knowledge Grid)**：
    *   將「嘉義木都」、「東市場」、「森林鐵路」等靜態知識模組化。
    *   作為 AI 回答之外的穩定參考資訊。
4.  **AI 互動問答區 (Interactive Q&A)**：
    *   網頁靈魂，使用多樣化表單 (`form`) 處理問題輸入。
    *   透過 API 與 Google Gemini 連結，實現即時導覽。
5.  **對話紀錄 (History)**：
    *   自動記錄使用者的問題與 AI 的回答，建立連貫的學習環境。
6.  **頁尾 (Footer)**：
    *   標註計畫團隊資訊、年份與技術來源。

## 三、 核心提示詞 (AI Prompts)

這是 AI 是否能像「導覽員」的關鍵。教學重點在於讓學生理解如何撰寫「角色設定 (Persona)」與「知識庫 (Knowledge Base)」。

### 提示詞策略範例：
```text
你是一位專業的嘉義木都文化導覽員。
1. [角色]：親切、適合國小高年級學生的口吻。
2. [知識庫]：載入木都由來、東市場卡榫智慧、森林鐵路技術、以及「校園碳險隊」科展研究數據（如 120 棵樹總固碳量 10.7 萬公斤等）。
3. [邏輯判斷]：如果問題無關（例如問數學題、或問台北歷史），請回答：『這個問題超過我的知識範圍。您可以問問：木都歷史、校園碳匯研究、檜木建築等話題喔！』
4. [知識擴張]：如果問題相關但知識庫沒提到，請根據導覽員的專業背景給予有趣的補充。
```

## 四、 實作練習建議 (Project Activities)

1.  **資料增補**：引導學生將自己學校的樹木調查數據加入知識庫，觀察 AI 回答的變化。
2.  **口吻修飾**：修改提示詞，讓 AI 變成「森林小精靈」或「老建築師」。
3.  **UI 字級優化**：針對視力不佳的長輩，調整 Tailwind 的 `text-base` 至 `text-lg`。

---

## 五、 API Key 規則與安全 (Secrets Management)

教學時務必強調 API Key 的安全管理，避免外洩。

1.  **不寫死程式碼 (No Hardcoding)**：
    *   絕對不要將 `AI_KEY_XXXX` 寫在程式碼中。
    *   應該存放在平台提供的 **Secrets 面板**。
2.  **環境變數傳遞 (Env Variables)**：
    *   在專案中透過 `process.env.GEMINI_API_KEY` 讀取。
    *   使用 Vite 的 `define` 配置將環境變數安全地注入前端。
3.  **金鑰連結 (Project Association)**：
    *   發佈 (Publish) 網站前，API Key 必須與一個 Google Cloud 專案關聯，這是一項安全管理的要求。

## 五、 設計風格指南 (Design System)

引導學生思考透過色彩與排版傳遞「文化質感」。

| 元素 | 設計選擇 | 目的 |
| :--- | :--- | :--- |
| **背景色** | `#fffaf0` (暖色調米白) | 減少純白刺眼感，增加紙質、溫柔的感覺。 |
| **主文字色** | `#5d4037` (木質咖啡色) | 直接聯覺木材，穩重且舒適。 |
| **主按鈕色** | `#2e7d32` (森林綠) | 代表植栽管理、阿里山森林與永續。 |
| **標題字體** | 粗體 Sans-serif | 在各設備(手機/電腦)上都有極佳的閱讀清晰度。 |
| **圓角半徑** | `3xl` (大幅圓角) | 消除尖銳邊緣，創造親切感與現代感。 |

## 六、 手機版特別優化 (Mobile Optimization)

手機是學生的主要設備，務必實作以下優化：
*   **Viewport 控制**：防止自動縮放影響閱讀。
*   **iOS 鍵盤優化**：輸入框設置 `inputMode="search"`。
*   **回應式設計**：在大螢幕顯示多欄位感路，在手機端自動垂直堆疊。

## 七、 同步到 GitHub 與發佈網站

若要將此專案同步到 GitHub 並讓其他人使用，請遵循以下步驟：

1.  **匯出專案 (Export)**：
    *   在開發環境右側選單點擊 **Settings (齒輪圖示)**。
    *   選擇 **Export to GitHub**。這會將目前的程式碼推送到您的 GitHub 倉庫（包括我為您準備的自動部署腳本）。
2.  **設定 GitHub Pages 部署方式 (靜態部署)**：
    *   在 GitHub 專案頁面點擊頂部的 **Settings** 分頁 -> 左側選單點擊 **Pages**。
    *   在 **Build and deployment** 下方的 **Source** 分類中，將原本的 `Deploy from a branch` 改為 **`GitHub Actions`**。
    *   **重要**：靜態部署 (GitHub Pages) 無法直接存取系統金鑰，因此訪客必須點擊「系統狀態」輸入個人金鑰方可使用。
3.  **Cloud Run/發佈版本 (全端部署 - 推薦)**：
    *   為了讓發佈後的網站能自動使用您設定的金鑰，我已將專案升級為**全端 (Full-Stack)** 架構。
    *   這意味著金鑰會存放在伺服器後端，安全性更高，且能解決 Cloud Run 上靜態部署無法讀取環境變數的問題。
    *   **設定方式**：請在部署面板或 Cloud Run 控制台的 **Environment Variables** (環境變數) 中新增 `GEMINI_API_KEY`，系統就會自動連通。
4.  **解決 404 錯誤與路徑問題**：
    *   我已經在 `vite.config.ts` 中加入了 `base: './'` 設定，這能確保網站在 GitHub Pages 的子目錄下也能正確載入圖片與腳本。
5.  **引導使用者輸入個人金鑰 (雙重保障)**：
    *   這是在 GitHub 上運作的最佳方式。請告知您的學生或訪客：
        1. 點擊標題旁的 「⚙️ 系統狀態」。
        2. 在 **「使用個人 API 金鑰」** 欄位貼上他們自己的 Gemini Key。
        3. 勾選 **「優先使用個人金鑰」**。
    *   **優點**：
        *   **隱私安全**：Key 不會經過您的伺服器，直接儲存在使用者本機的 `localStorage` 中執行。
        *   **零成本**：您不需要負擔所有使用者的 API 額度。
        *   **教學意義**：讓學生學習如何管理自己的 AI 資源。
